# Обобщённая теория балансных алгебр

## 1. Мотивация

В базовой доменной теории баланс определяется как сумма вкладов транзакций:

$$B_F(p) = \sum_{\substack{e \in E,\ e \in D,\ Scope_R(e)=1,\\ BalPath_F(e) = p}} Val(e).$$

Сумма — это конкретный выбор функции агрегации. Из этого выбора следует свойство, которое в базовой теории называется «аддитивность»:

$$B^{E_1 \cup E_2}(p) = B^{E_1}(p) + B^{E_2}(p).$$

Но аддитивность — не аксиома модели. Это **следствие** выбора суммирования в качестве функции агрегации. Другая функция порождает другое правило связи между частями и целым.

Цель данного документа — определить **класс допустимых функций агрегации** (балансных алгебр), показать, что каждая такая функция порождает собственное правило связи «родитель — потомки», и выяснить, какие свойства доменной модели являются действительно фундаментальными, а какие — частными следствиями выбора суммы.

---

## 2. Центральное наблюдение

Рассмотрим дерево или roll-up, в котором грубый ключ $k$ раскрывается в детальные ключи $k_1, \dots, k_m$.

В базовой теории значение родителя определяется через детей:

$$B(k) = \sum_{i=1}^{m} B(k_i).$$

Это конкретное правило связи: **значение целого равно сумме частей**.

Но если функция агрегации — среднее, то правило связи другое:

$$B(k) = \frac{\sum_i n_i \cdot B(k_i)}{\sum_i n_i},$$

где $n_i$ — число транзакций в $i$-м потомке. Значение целого — это **взвешенное среднее** частей, а не их сумма.

Если функция — максимум:

$$B(k) = \max_i B(k_i).$$

Значение целого — это **максимум** по частям.

Таким образом, **выбор функции агрегации порождает правило связи между родителем и потомками**. Аддитивность — это правило связи, порождённое суммой. Для других функций — другие правила.

Вопрос: что общего у этих правил? Какая структура объединяет сумму, среднее, максимум и другие допустимые функции?

## 3. Балансная алгебра

### 3.1. Определение

Балансная алгебра — это кортеж

$$\mathcal A = (S,\ s_0,\ \oplus,\ inc,\ val),$$

где:

- $S$ — пространство внутренних состояний;
- $s_0 \in S$ — начальное состояние (пустой агрегат);
- $\oplus : S \times S \to S$ — операция слияния двух состояний;
- $inc : S \times \mathbb R \to S$ — операция включения нового значения транзакции в состояние;
- $val : S \to \mathbb R$ — функция извлечения итогового значения из состояния.

### 3.2. Аксиомы

Операция слияния $\oplus$ удовлетворяет:

1. **Ассоциативность:** $(s_1 \oplus s_2) \oplus s_3 = s_1 \oplus (s_2 \oplus s_3)$
2. **Коммутативность:** $s_1 \oplus s_2 = s_2 \oplus s_1$
3. **Нейтральный элемент:** $s_0 \oplus s = s$

Включение согласовано со слиянием:

4. **Совместимость:** $inc(s, x) = s \oplus inc(s_0, x)$

### 3.3. Интерпретация

$(S, \oplus, s_0)$ — коммутативный моноид. Это и есть фундаментальная структура, которая объединяет все допустимые функции агрегации.

Смысл аксиом:

- ассоциативность и коммутативность гарантируют, что **порядок обработки транзакций не влияет на результат**;
- нейтральный элемент гарантирует, что **пустое множество транзакций даёт определённое начальное состояние**;
- совместимость гарантирует, что **добавить транзакцию — то же самое, что слить с одноэлементным состоянием**.

### 3.4. Два уровня: состояние и значение

Ключевое различие обобщённой теории от базовой:

- **Состояние** $\sigma \in S$ — это то, что хранится. Для суммы это одно число, для среднего — пара (сумма, количество), для дисперсии — тройка.
- **Значение** $val(\sigma) \in \mathbb R$ — это то, что читается. Для суммы $val = id$, для среднего $val((s, n)) = s/n$.

Слияние работает на уровне состояний. Извлечение значения — последний шаг, применяемый при чтении.

## 4. Правило связи «родитель — потомки»

### 4.1. Общая формулировка

Для любой балансной алгебры правило связи между родительским ключом $k$ и его потомками $k_1, \dots, k_m$ определяется через слияние состояний:

$$\sigma(k) = \sigma(k_1) \oplus \sigma(k_2) \oplus \dots \oplus \sigma(k_m).$$

Итоговое значение родителя:

$$B(k) = val(\sigma(k)) = val\left(\bigoplus_{i=1}^{m} \sigma(k_i)\right).$$

Это и есть **обобщённое балансное уравнение**. Оно утверждает: состояние целого равно слиянию состояний частей.

### 4.2. Частные случаи

**Сумма.** $S = \mathbb R$, $\oplus = +$, $val = id$.

$$B(k) = val\left(\sum_i \sigma(k_i)\right) = \sum_i B(k_i).$$

Правило связи: значение родителя равно сумме значений потомков. Это и есть «аддитивность» базовой теории.

**Среднее.** $S = \mathbb R \times \mathbb N$, $\oplus((s_1,n_1),(s_2,n_2)) = (s_1+s_2, n_1+n_2)$, $val((s,n)) = s/n$.

$$\sigma(k) = \left(\sum_i s_i,\ \sum_i n_i\right)$$

$$B(k) = \frac{\sum_i s_i}{\sum_i n_i} = \frac{\sum_i n_i \cdot B(k_i)}{\sum_i n_i}.$$

Правило связи: значение родителя — взвешенное среднее потомков, где веса — количества транзакций.

**Максимум.** $S = \mathbb R \cup \{-\infty\}$, $\oplus = \max$, $val = id$.

$$B(k) = \max_i B(k_i).$$

Правило связи: значение родителя — максимум по потомкам.

### 4.3. Почему правило следует из выбора алгебры

Правило связи не задаётся отдельно. Оно **выводится** из операции $\oplus$ и функции $val$. Зная алгебру, мы автоматически знаем:

- как состояния агрегируются снизу вверх;
- как итоговое значение родителя выражается через значения (или состояния) потомков;
- какие тождества выполняются.

Аддитивность — это тождество $val(\sigma_1 \oplus \sigma_2) = val(\sigma_1) + val(\sigma_2)$, которое истинно для SUM и ложно для AVG.

## 5. Сливаемость как фундаментальное свойство

### 5.1. Определение

Сливаемость (mergeability) — это свойство, общее для **всех** балансных алгебр:

$$\sigma^{E_1 \cup E_2}(p) = \sigma^{E_1}(p) \oplus \sigma^{E_2}(p).$$

Состояние, построенное по объединению множеств транзакций, равно слиянию состояний, построенных по каждому множеству отдельно.

### 5.2. Сливаемость vs. аддитивность

Аддитивность — частный случай сливаемости для алгебры SUM:

$$B^{E_1 \cup E_2}(p) = B^{E_1}(p) + B^{E_2}(p).$$

Это работает, потому что для SUM $val = id$ и $\oplus = +$, поэтому сливаемость на уровне состояний **совпадает** с аддитивностью на уровне значений.

Для AVG сливаемость на уровне состояний сохраняется:

$$(s_1, n_1) \oplus (s_2, n_2) = (s_1 + s_2, n_1 + n_2),$$

но на уровне значений аддитивность **не** выполняется:

$$\frac{s_1 + s_2}{n_1 + n_2} \ne \frac{s_1}{n_1} + \frac{s_2}{n_2}.$$

### 5.3. Что даёт сливаемость

Сливаемость гарантирует:

- **параллельную обработку:** можно разбить транзакции на части, обработать независимо, затем слить состояния;
- **инкрементальное обновление:** новая транзакция — это слияние текущего состояния с одноэлементным;
- **корректный roll-up:** состояние грубого ключа равно слиянию состояний детальных ключей;
- **восстановление после сбоя:** состояния можно пересчитать по любому подмножеству и слить.

Это **фундаментальное** свойство модели. Аддитивность — его частное проявление.

## 6. Прозрачные и непрозрачные алгебры

### 6.1. Определение

Балансная алгебра называется **прозрачной**, если $val$ — тождество, то есть $S = \mathbb R$ и $val(s) = s$.

Балансная алгебра называется **непрозрачной**, если $val$ — нетривиальная функция, то есть состояние и итоговое значение различаются.

### 6.2. Ключевое различие

Для прозрачных алгебр правило связи «родитель — потомки» выражается **через итоговые значения**:

$$B(k) = B(k_1) \oplus B(k_2) \oplus \dots \oplus B(k_m).$$

Для непрозрачных алгебр правило связи работает только **на уровне состояний**. Значение родителя нельзя вычислить из значений потомков — нужны их состояния.

### 6.3. Примеры

| Алгебра | Прозрачность | $S$ | $val$ | Правило связи на уровне значений |
|---|---|---|---|---|
| SUM | прозрачная | $\mathbb R$ | $id$ | $B(k) = \sum_i B(k_i)$ |
| COUNT | прозрачная | $\mathbb N$ | $id$ | $B(k) = \sum_i B(k_i)$ |
| MIN | прозрачная | $\mathbb R \cup \{+\infty\}$ | $id$ | $B(k) = \min_i B(k_i)$ |
| MAX | прозрачная | $\mathbb R \cup \{-\infty\}$ | $id$ | $B(k) = \max_i B(k_i)$ |
| AVG | непрозрачная | $\mathbb R \times \mathbb N$ | $s/n$ | невозможно без знания $(s_i, n_i)$ |
| WEIGHTED AVG | непрозрачная | $\mathbb R \times \mathbb R$ | $s/w$ | невозможно без знания $(s_i, w_i)$ |
| VAR | непрозрачная | $\mathbb R \times \mathbb R \times \mathbb N$ | $q/n - (s/n)^2$ | невозможно без знания $(s_i, q_i, n_i)$ |

### 6.4. Бизнес-смысл

Для прозрачных алгебр клиент может работать только с итоговыми значениями. Зная GMV по странам, можно вычислить GMV по региону — просто сложить.

Для непрозрачных алгебр клиенту нужен доступ к состояниям, чтобы корректно агрегировать. Зная средний чек по странам, нельзя вычислить средний чек по региону — нужны суммы и количества.

## 7. Конкретные алгебры

### 7.1. SUM

$$\mathcal A_{sum} = (\mathbb R,\ 0,\ +,\ inc,\ id)$$

- $inc(s, x) = s + x$
- Правило связи: $B(k) = \sum_i B(k_i)$

Базовая алгебра бухгалтерского баланса. Баланс счёта — сумма всех проводок.

### 7.2. COUNT

$$\mathcal A_{count} = (\mathbb N,\ 0,\ +,\ inc,\ id)$$

- $inc(s, x) = s + 1$ (значение транзакции игнорируется)
- Правило связи: $B(k) = \sum_i B(k_i)$

Число заказов, число транзакций. Формально SUM и COUNT порождают одинаковое правило связи (сумма), но различаются функцией $inc$.

### 7.3. AVG

$$\mathcal A_{avg} = (\mathbb R \times \mathbb N,\ (0,0),\ \oplus,\ inc,\ val)$$

- $\oplus((s_1,n_1),(s_2,n_2)) = (s_1+s_2,\ n_1+n_2)$
- $inc((s,n), x) = (s+x,\ n+1)$
- $val((s,n)) = s/n$

Правило связи:

$$B(k) = \frac{\sum_i n_i \cdot B(k_i)}{\sum_i n_i}$$

Средний чек, средний GMV заказа, средняя стоимость поездки.

### 7.4. Взвешенное среднее (WEIGHTED AVG)

$$\mathcal A_{wavg} = (\mathbb R \times \mathbb R,\ (0,0),\ \oplus,\ inc,\ val)$$

- $\oplus((s_1,w_1),(s_2,w_2)) = (s_1+s_2,\ w_1+w_2)$
- $inc((s,w_{total}), (x, w)) = (s + w \cdot x,\ w_{total} + w)$
- $val((s,w_{total})) = s / w_{total}$

Правило связи:

$$B(k) = \frac{\sum_i w_i \cdot B(k_i)}{\sum_i w_i}$$

Средневзвешенный курс валюты, средневзвешенная цена портфеля.

### 7.5. MIN / MAX

$$\mathcal A_{min} = (\mathbb R \cup \{+\infty\},\ +\infty,\ \min,\ inc,\ id)$$

$$\mathcal A_{max} = (\mathbb R \cup \{-\infty\},\ -\infty,\ \max,\ inc,\ id)$$

- $inc(s, x) = \min(s, x)$ или $\max(s, x)$
- Правило связи: $B(k) = \min_i B(k_i)$ или $B(k) = \max_i B(k_i)$

Экстремальные значения для контроля рисков.

### 7.6. Дисперсия (VAR)

$$\mathcal A_{var} = (\mathbb R \times \mathbb R \times \mathbb N,\ (0,0,0),\ \oplus,\ inc,\ val)$$

- Состояние $(sum, sum\_sq, count)$
- $\oplus((s_1,q_1,n_1),(s_2,q_2,n_2)) = (s_1+s_2,\ q_1+q_2,\ n_1+n_2)$
- $inc((s,q,n), x) = (s+x,\ q+x^2,\ n+1)$
- $val((s,q,n)) = q/n - (s/n)^2$

Трёхмерное состояние. Волатильность транзакций, разброс чеков.

## 8. Обобщённое семейство балансов

### 8.1. Определение

В обобщённой теории семейство балансов:

$$F = (D, R, M, \Theta, \mathcal A).$$

Пять компонент:

- $D$ — домен (класс аддитивности, а точнее — класс сливаемости);
- $R$ — правило (scope);
- $M$ — материализация (groupBy);
- $\Theta$ — темпоральная спецификация;
- $\mathcal A$ — балансная алгебра.

### 8.2. Balance row

Balance row хранит состояние $\sigma_F(p) \in S$, а не число.

Итоговое значение:

$$B_F(p) = val(\sigma_F(p)).$$

### 8.3. Запись транзакции

При поступлении транзакции $e$, попадающей в путь $p$:

$$\sigma_F(p) \leftarrow inc(\sigma_F(p),\ Val(e)).$$

Сложность: $O(k)$ по числу активных семейств — как в базовой теории.

### 8.4. Чтение

$$Read(p) = val(\sigma_F(p)).$$

Сложность: $O(1)$ — как в базовой теории.

## 9. Roll-up в обобщённой модели

### 9.1. Формулировка

Roll-up определяется на уровне состояний:

$$\sigma_{coarse}(k) = \bigoplus_{k' \to k} \sigma_{fine}(k').$$

Значение:

$$B_{coarse}(k) = val\left(\bigoplus_{k' \to k} \sigma_{fine}(k')\right).$$

### 9.2. Пример: roll-up среднего чека

Детальные строки:

| ключ | состояние (sum, count) | значение (avg) |
|---|---|---|
| (EE, Food) | (1 000, 50) | 20.00 |
| (EE, Grocery) | (600, 20) | 30.00 |

Наивный roll-up по значениям: $(20 + 30) / 2 = 25$ — **неверно**.

Корректный roll-up по состояниям:

$$\sigma(EE) = (1000, 50) \oplus (600, 20) = (1600, 70)$$

$$B(EE) = val((1600, 70)) = 1600 / 70 \approx 22.86$$

### 9.3. Пример: roll-up максимума

| ключ | значение (max) |
|---|---|
| (EE, Food) | 12 000 |
| (EE, Grocery) | 8 000 |

$$B(EE) = \max(12\,000, 8\,000) = 12\,000$$

Для прозрачных алгебр roll-up по значениям корректен. Для непрозрачных — только по состояниям.

## 10. Conversion в обобщённой модели

### 10.1. Conversion внутри одной алгебры

Если оба семейства используют одну алгебру, conversion работает как в базовой теории:

$$\sigma_{target}(p_B) = \bigoplus_{target} \{\Gamma_S(\sigma_{source}(p_A)) : \Psi(p_A) = p_B\},$$

где $\Gamma_S : S \to S$ — преобразование состояний.

### 10.2. Conversion между алгебрами

Conversion $\mathcal A_1 \to \mathcal A_2$ требует отображения $\Gamma_S : S_1 \to S_2$, согласованного со слиянием:

$$\Gamma_S(s_1 \oplus_1 s_2) = \Gamma_S(s_1) \oplus_2 \Gamma_S(s_2).$$

Такое отображение — гомоморфизм моноидов.

### 10.3. Информационное ограничение

Conversion возможен только если состояние источника содержит достаточно информации для построения состояния цели.

Допустимые направления:

- AVG → SUM: $\Gamma_S((s, n)) = s$ — из среднего можно извлечь сумму.
- AVG → COUNT: $\Gamma_S((s, n)) = n$ — из среднего можно извлечь количество.
- VAR → AVG: $\Gamma_S((s, q, n)) = (s, n)$ — из дисперсии можно извлечь среднее.

Недопустимые направления:

- SUM → AVG: из суммы нельзя восстановить количество.
- COUNT → SUM: из количества нельзя восстановить сумму.
- MAX → SUM: из максимума нельзя восстановить сумму.

### Бизнес-смысл

Если система хранит средний чек (состояние = сумма + количество), из него можно получить GMV (сумму) и количество заказов. Но если хранится только GMV (одно число), средний чек получить нельзя — потеряна информация о количестве.

Это фундаментальный принцип: **более богатое состояние содержит более бедное, но не наоборот**.

## 11. Derived composition в обобщённой модели

### 11.1. Формулировка

Derived composition работает на уровне **извлечённых значений**:

$$B_{F_*}(p) = \Phi(val_1(\sigma_{F_1}(p)), \dots, val_n(\sigma_{F_n}(p))).$$

### 11.2. Пример: profit = income − expense

Оба семейства используют SUM. $val = id$.

$$B_{profit}(p) = B_{income}(p) - B_{expense}(p).$$

### 11.3. Пример: revenue per order = GMV / order count

Семейство GMV использует SUM. Семейство order count использует COUNT.

$$B_{rpo}(p) = \frac{val_{sum}(\sigma_{GMV}(p))}{val_{count}(\sigma_{orders}(p))}.$$

Результат — число, не имеющее собственной алгебры. Derived family — это производный объект, он вычисляется, а не накапливается.

### 11.4. Инвариант: несколько алгебр в одном домене

Если в одном домене одновременно живут SUM, COUNT и AVG с одинаковым scope и groupBy, между ними выполняется тождество:

$$val_{sum}(\sigma_{F_{sum}}(p)) = val_{avg}(\sigma_{F_{avg}}(p)) \times val_{count}(\sigma_{F_{count}}(p)),$$

то есть: GMV = средний чек × количество заказов.

Это не случайное совпадение, а следствие того, что все три семейства работают с одним потоком транзакций.

## 12. Идемпотентность

### 12.1. Определение

Алгебра называется **идемпотентной**, если:

$$s \oplus s = s \quad \forall s \in S.$$

### 12.2. Примеры

- MIN, MAX — идемпотентны: $\min(x, x) = x$, $\max(x, x) = x$.
- SUM, COUNT, AVG — неидемпотентны: $s + s = 2s \ne s$.

### 12.3. Практическое значение

Идемпотентные алгебры безопасны при дублировании транзакций: повторная обработка одной и той же транзакции не изменяет результат. Неидемпотентные требуют гарантии exactly-once.

Это важно при проектировании write path: если система допускает at-least-once доставку транзакций, идемпотентные алгебры корректны без дедупликации.

## 13. Темпоральная совместимость

Темпоральное расширение из balance-domains-temporal полностью совместимо с обобщёнными алгебрами.

Для семейства $F = (D, R, M, turn(g), \mathcal A)$ запись транзакции $e$ с $\tau(e)$ обновляет:

$$\sigma_F(k_M, \omega) \leftarrow inc(\sigma_F(k_M, \omega),\ Val(e)),$$

где $\omega = \pi_g(\tau(e))$.

Temporal roll-up:

$$\sigma_{month}(k_M, m) = \bigoplus_{d \in days(m)} \sigma_{day}(k_M, d).$$

Accumulated:

$$\sigma_{acc}(k_M) = \bigoplus_{\omega} \sigma_{turn}(k_M, \omega).$$

Все формулы идентичны базовой теории, только $+$ заменён на $\oplus$.

---

## 14. Практические примеры

### 14.1. Средний чек по стране и месяцу

Домен: `trx / EUR`. Rule: completed orders. Materialization: `groupBy = (country)`, $\Theta = turn(month)$, $\mathcal A = \mathcal A_{avg}$.

Транзакции за январь 2025:

| дата | country | amount |
|---|---|---|
| 2025-01-05 | EE | 15 |
| 2025-01-10 | EE | 25 |
| 2025-01-18 | EE | 20 |
| 2025-01-22 | LV | 30 |
| 2025-01-28 | LV | 10 |

Результат:

| country | month | состояние (sum, count) | значение (avg) |
|---|---|---|---|
| EE | 2025-01 | (60, 3) | 20.00 |
| LV | 2025-01 | (40, 2) | 20.00 |

Roll-up «все страны»:

$$\sigma = (60, 3) \oplus (40, 2) = (100, 5), \quad B = 20.00$$

### 14.2. Средневзвешенный курс конвертации

Домен: `trx / EUR`. Rule: currency conversions. $\mathcal A = \mathcal A_{wavg}$.

| дата | source | rate | volume |
|---|---|---|---|
| 2025-01-05 | USD | 1.08 | 10 000 |
| 2025-01-12 | USD | 1.10 | 5 000 |
| 2025-01-20 | USD | 1.07 | 15 000 |

$$\sigma = (1.08 \times 10000 + 1.10 \times 5000 + 1.07 \times 15000,\ 30000) = (32\,350,\ 30\,000)$$

$$B = 32\,350 / 30\,000 \approx 1.0783$$

Наивное среднее: $(1.08 + 1.10 + 1.07)/3 \approx 1.0833$ — неверно.

### 14.3. Несколько алгебр в одном домене

| семейство | алгебра | бизнес-смысл |
|---|---|---|
| $F_1$ | SUM | GMV по стране |
| $F_2$ | COUNT | число заказов |
| $F_3$ | AVG | средний чек |
| $F_4$ | MAX | максимальный заказ |

Одна транзакция $e$ с $Val(e) = 500$ обновляет все четыре:

- $F_1$: $\sigma \leftarrow \sigma + 500$
- $F_2$: $\sigma \leftarrow \sigma + 1$
- $F_3$: $\sigma \leftarrow (s + 500, n + 1)$
- $F_4$: $\sigma \leftarrow \max(\sigma, 500)$

Инвариант: $val_{F_1} = val_{F_3} \times val_{F_2}$.

## 15. Нетривиальные балансные алгебры

Предыдущие примеры (SUM, AVG, MAX) интуитивно просты. Однако класс балансных алгебр значительно шире. Ниже — реальные примеры, каждый из которых удовлетворяет аксиомам §3.2 и имеет прямое практическое применение.

### 15.1. HyperLogLog (приближённый COUNT DISTINCT)

**Задача:** сколько уникальных пользователей совершили транзакции по стране за месяц.

Точный подсчёт уникальных значений требует хранения множества всех идентификаторов — размер состояния растёт линейно с числом уникальных элементов. HyperLogLog (HLL) — это вероятностная структура фиксированного размера (~12 КБ), оценивающая кардинальность с погрешностью ~2%.

$$\mathcal A_{hll} = (\mathbb Z^m,\ \mathbf{0},\ \oplus,\ inc,\ val)$$

- $S = \mathbb Z^m$ — вектор из $m$ регистров (обычно $m = 2^{14} = 16384$);
- $s_0 = \mathbf{0}$ — нулевой вектор;
- $\oplus(\mathbf{r}_1, \mathbf{r}_2) = (\max(r_{1,1}, r_{2,1}), \dots, \max(r_{1,m}, r_{2,m}))$ — поэлементный максимум;
- $inc(\mathbf{r}, x)$: вычисляет хеш $x$, определяет индекс регистра и обновляет его;
- $val(\mathbf{r})$: гармоническое среднее по регистрам, скорректированное поправочным коэффициентом.

**Проверка аксиом:**

- Ассоциативность: $\max$ ассоциативен.
- Коммутативность: $\max$ коммутативен.
- Нейтральный элемент: $\max(\mathbf{0}, \mathbf{r}) = \mathbf{r}$.
- Совместимость: включение элемента $x$ в состояние $\mathbf{r}$ эквивалентно слиянию $\mathbf{r}$ с одноэлементным скетчем.

**Roll-up.** Слияние HLL-скетчей по странам даёт HLL-скетч по региону, корректно оценивающий число уникальных пользователей **с учётом пересечений** (пользователь, активный в EE и LV, посчитается один раз).

**Классификация:** непрозрачная ($val$ — нетривиальная функция), идемпотентная ($\mathbf{r} \oplus \mathbf{r} = \mathbf{r}$, поскольку $\max(r_i, r_i) = r_i$).

### Бизнес-смысл

Число уникальных клиентов по стране, число уникальных мерчантов, число уникальных устройств. Всё это — COUNT DISTINCT, и HLL позволяет вести его как balance row с фиксированным размером состояния и корректным roll-up.

---

### 15.2. Гистограмма (распределение по бакетам)

**Задача:** распределение чеков по диапазонам (0–10, 10–50, 50–200, 200+) по стране и месяцу.

$$\mathcal A_{hist} = (\mathbb N^b,\ \mathbf{0},\ +,\ inc,\ val)$$

- $S = \mathbb N^b$ — вектор из $b$ счётчиков, где $b$ — число бакетов;
- $s_0 = \mathbf{0}$;
- $\oplus = +$ (поэлементное сложение);
- $inc(\mathbf{h}, x)$: определяет бакет $j$ для значения $x$ и увеличивает $h_j$ на 1;
- $val(\mathbf{h})$: зависит от задачи — может вернуть весь вектор, моду, приближённый перцентиль или долю.

**Проверка аксиом:** поэлементное сложение — коммутативный моноид с нейтральным элементом $\mathbf{0}$.

**Roll-up.** Гистограмма по Балтике — поэлементная сумма гистограмм по странам. Это точная операция, не приближённая.

**Классификация:** непрозрачная (состояние — вектор, значение — скаляр или интерпретация вектора), неидемпотентная.

### Бизнес-смысл

Распределение чеков позволяет видеть не только средний чек, но и его структуру: какая доля заказов мелкие, средние, крупные. Это невозможно получить из SUM или AVG — нужна гистограмма.

Пример: GMV по Эстонии одинаковый в январе и феврале, но в январе 80% чеков мелкие, а в феврале — 50% крупных. SUM и AVG этого не покажут, гистограмма покажет.

---

### 15.3. T-Digest (приближённые квантили)

**Задача:** 95-й перцентиль времени доставки по стране и месяцу.

T-Digest — структура из ~200–300 центроидов, аппроксимирующая распределение значений.

$$\mathcal A_{tdigest} = (C,\ \emptyset,\ \oplus,\ inc,\ val_q)$$

- $S = C$ — множество центроидов (пар «среднее, вес»), ограниченное по размеру;
- $s_0 = \emptyset$ — пустое множество;
- $\oplus(C_1, C_2)$: объединение центроидов с последующим сжатием до максимального размера;
- $inc(C, x)$: добавление центроида $(x, 1)$ с последующим сжатием;
- $val_q(C)$: оценка $q$-го квантиля по центроидам.

**Аксиомы:** выполняются приближённо. Слияние ассоциативно и коммутативно с точностью до порядка сжатия. На практике погрешность контролируема и составляет ~0.5–1% для экстремальных квантилей (p95, p99).

**Roll-up.** Слияние t-digest по вертикалям даёт t-digest по стране, из которого извлекается приближённый перцентиль.

**Классификация:** непрозрачная, неидемпотентная, приближённая.

### Бизнес-смысл

SLA по времени доставки: «95% заказов доставлены менее чем за 30 минут». Для контроля этого SLA нужен p95, а не среднее. T-Digest позволяет вести p95 как balance row с корректным roll-up.

---

### 15.4. Bloom Filter (приближённая проверка принадлежности)

**Задача:** встречался ли данный `merchant_id` среди транзакций по стране за месяц.

$$\mathcal A_{bloom} = (\{0,1\}^m,\ \mathbf{0},\ \lor,\ inc,\ val)$$

- $S = \{0,1\}^m$ — битовый вектор длины $m$;
- $s_0 = \mathbf{0}$;
- $\oplus = \lor$ (побитовое OR);
- $inc(\mathbf{b}, x)$: устанавливает биты $h_1(x), \dots, h_k(x)$ в 1, где $h_i$ — хеш-функции;
- $val(\mathbf{b})$: проверка принадлежности — все биты $h_1(x), \dots, h_k(x)$ установлены?

**Проверка аксиом:** побитовое OR — коммутативный, ассоциативный, с нейтральным элементом $\mathbf{0}$.

**Roll-up.** OR битовых векторов по странам даёт фильтр по региону. Если merchant встречался хотя бы в одной стране, он будет обнаружен в региональном фильтре.

**Классификация:** непрозрачная, идемпотентная ($\mathbf{b} \lor \mathbf{b} = \mathbf{b}$).

### Бизнес-смысл

Быстрая проверка: работал ли мерчант в данной стране в данном месяце? Без полного сканирования транзакций. Полезно для compliance, fraud detection, быстрой фильтрации.

---

### 15.5. Ковариация и корреляция двух мер

**Задача:** корреляция между GMV и временем доставки по стране.

Каждая транзакция несёт две меры: $x$ (GMV) и $y$ (время доставки).

$$\mathcal A_{cov} = (\mathbb R^5,\ \mathbf{0},\ +,\ inc,\ val)$$

- Состояние: $(s_x, s_y, s_{xx}, s_{yy}, s_{xy}, n)$ — суммы, суммы квадратов, сумма произведений, счётчик;
- $\oplus$ — поэлементное сложение;
- $inc$: обновляет все шесть компонент;
- $val$: $cov = s_{xy}/n - (s_x/n)(s_y/n)$, $corr = cov / (\sigma_x \cdot \sigma_y)$.

**Проверка аксиом:** поэлементное сложение — коммутативный моноид.

**Roll-up.** Ковариация по региону вычисляется корректно через слияние шестимерных состояний.

### Бизнес-смысл

Есть ли связь между размером заказа и временем доставки? Растёт ли время доставки с ростом GMV? Это аналитический вопрос, ответ на который можно вести как balance row.

## 16. Что не входит в класс балансных алгебр

### 16.1. Скользящее среднее (Moving Average)

Скользящее среднее по последним $k$ транзакциям требует упорядоченности и конечного окна. Оно не коммутативно: порядок транзакций существенен. Не является коммутативным моноидом.

### 16.2. Точные квантили и медианы

Точное вычисление медианы требует хранения всех значений — бесконечномерное состояние. T-Digest и KLL sketch (§15.3) являются приближёнными балансными алгебрами с контролируемой погрешностью, но точные квантили — нет.

### 16.3. Условные агрегаты (COUNT IF, SUM IF)

Это не новые алгебры, а комбинация scope и существующих алгебр. COUNT IF(country = EE) — это просто семейство с scope `country = EE` и алгеброй COUNT.

## 17. Иерархия свойств модели

### 17.1. Фундаментальные свойства (не зависят от выбора алгебры)

- Доменная изоляция
- Непересекаемость доменов
- Scope правила
- Materialization (groupBy)
- Path-space-disjointness
- Темпоральная спецификация
- Инкрементальность записи ($O(k)$)
- Чтение по каноническому пути ($O(1)$)
- **Сливаемость** ($\sigma^{E_1 \cup E_2} = \sigma^{E_1} \oplus \sigma^{E_2}$)

### 17.2. Свойства, зависящие от алгебры

- Правило связи «родитель — потомки»: определяется через $\oplus$ и $val$
- Roll-up на уровне значений: только для прозрачных алгебр
- Аддитивность: только для SUM
- Идемпотентность: только для MIN, MAX
- Допустимые conversion: определяются существованием гомоморфизма моноидов

### 17.3. Итог

Базовая доменная теория — это обобщённая теория с $\mathcal A = \mathcal A_{sum}$. Аддитивность — следствие выбора суммы. Фундаментальное свойство — сливаемость, а не аддитивность.
