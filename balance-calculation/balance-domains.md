# Формальная теория доменов, правил, материализаций и текущих балансов

## 1. Назначение модели

Модель описывает систему, которая по потоку транзакций ведёт текущие балансы в дискретном пространстве измерений.

Модель должна обеспечивать:

- строгую изоляцию аддитивности;
- однозначную принадлежность транзакции домену;
- однозначную материализацию балансов по каноническим путям;
- чтение баланса за $O(1)$ по полному каноническому пути;
- возможность конфигурировать как бухгалтерские balances, так и прикладные агрегаты, например GMV;
- возможность строить преобразования и производные balances поверх уже определённых семейств балансов.

Базовая идея модели такова:

1. каждая транзакция имеет полный канонический путь в общем пространстве измерений;
2. каждая транзакция принадлежит ровно одному аддитивному домену;
3. внутри этого домена применяются правила;
4. каждое правило вместе с materialization порождает семейство балансов;
5. каждая строка баланса адресуется полным каноническим путём;
6. значение строки баланса равно сумме вкладов всех транзакций, попавших в этот путь.

### Бизнес-смысл

Одна и та же экономическая операция может участвовать в разных балансах и отчётах, но только внутри точно определённого класса аддитивности.

Например, система может различать:

- transaction-basis EUR,
- transaction-basis USD,
- reporting-basis EUR.

Это разные классы суммы, и они не должны смешиваться автоматически.
Внутри одного такого класса можно вести:

- бухгалтерский баланс по счёту,
- баланс по стране,
- баланс по legal entity,
- GMV по вертикали,
- иные материализованные balances.

---

## 2. Пространство измерений

Пусть задано конечное множество измерений
$$\mathbb D = \{d_1,\dots,d_n\}.$$

Для каждого измерения $d \in \mathbb D$ задано множество допустимых значений
$$Val(d).$$

На множестве $\mathbb D$ фиксирован единый глобальный порядок
$$d_1 \prec d_2 \prec \dots \prec d_n.$$

Этот порядок используется для построения всех канонических путей.

### Бизнес-смысл

$\mathbb D$ — это полный словарь координат системы. Например:

- `basis`
- `currency`
- `account_type`
- `account_subtype`
- `country`
- `vertical`
- `legal_entity`
- `participant`
- `aging_bucket`

Единый порядок нужен, чтобы один и тот же баланс нельзя было адресовать двумя разными способами.

---

## 3. Транзакция

Транзакция $e$ — это частичная функция на пространстве измерений:
$$e : \mathbb D \rightharpoonup \bigcup_{d\in\mathbb D} Val(d).$$

Кроме значений измерений, транзакция несёт числовой вклад, задаваемый функцией
$$Val(e) \in \mathbb R.$$

На текущем этапе $Val(e)$ рассматривается как абстрактная числовая функция, возвращающая одно число.

### Бизнес-смысл

Транзакция — это элементарное событие учёта.

Пример бухгалтерской транзакции:

- `basis = trx`
- `currency = EUR`
- `account_type = expense`
- `account_subtype = ops_cost`
- `country = EE`
- `legal_entity = Company Ops OÜ`
- `Val(e) = 120`

Пример GMV-события:

- `basis = trx`
- `currency = EUR`
- `country = LV`
- `vertical = Food`
- `Val(e) = 45`

Модель не фиксирует заранее, что именно представляет $Val(e)$. Это может быть:

- signed accounting amount,
- GMV amount,
- fee amount,
- net amount.

---

## 4. Полный канонический путь транзакции

Полный канонический путь транзакции $e$, обозначаемый
$$Path(e),$$
строится по всем измерениям системы $\mathbb D$ в глобальном порядке.

Для каждого измерения $d_i$:

- если $e(d_i)$ определено, в путь включается соответствующий сегмент;
- если $e(d_i)$ не определено, сегмент остаётся пустым или отсутствующим в зависимости от реализации.

В теории достаточно считать, что $Path(e)$ — это полное упорядоченное описание транзакции по всем измерениям системы.

### Бизнес-смысл

Это полный адрес транзакции в системе.

Если транзакция описана как:

- `basis = trx`
- `currency = EUR`
- `account_type = expense`
- `account_subtype = ops_cost`
- `country = EE`
- `vertical = Food`

то её полный путь — это все эти значения, записанные в одном стандартном порядке.

Это ещё не баланс и не агрегат. Это полная координатная запись исходного события, из которой дальше будут выделяться:

- доменный префикс,
- scope правила,
- materialization key.

---

## 5. Аддитивный домен

### 5.1. Интуиция

Аддитивный домен задаёт максимальный класс транзакций, которые допустимо складывать вместе.

Именно домен изолирует аддитивность.

Каждая транзакция должна принадлежать ровно одному активному домену.

---

### 5.2. Домен как шаблон канонического префикса

Пусть домен $D$ задан на начальном сегменте измерений
$$(d_1,\dots,d_r), \qquad r \le n.$$

Для каждого измерения $d_i$ домен задаёт допустимое множество значений
$$S_i \subseteq Val(d_i).$$

Тогда домен определяется как шаблон канонического префикса:
$$D = ((d_1,S_1),\dots,(d_r,S_r)).$$

Этот шаблон называется доменным каноническим префиксом и обозначается
$$DomPath(D).$$

### Бизнес-смысл

Домен фиксирует класс суммы.

Например, можно определить домены:

1. `trx + EUR`
   $$D_1 = \{ basis = trx,\ currency = EUR \}$$

2. `trx + USD`
   $$D_2 = \{ basis = trx,\ currency = USD \}$$

3. `rep + EUR`
   $$D_3 = \{ basis = rep,\ currency = EUR \}$$

Это означает, что:

- transaction-basis EUR,
- transaction-basis USD,
- reporting-basis EUR

живут в разных доменах и не смешиваются автоматически.

---

### 5.3. Принадлежность транзакции домену

Транзакция $e$ принадлежит домену $D$, если её полный путь согласован с доменным префиксом.

Формально:
$$e \in D
\iff
\forall i=1,\dots,r:\ e(d_i)\text{ определено и } e(d_i)\in S_i.$$

### Объяснение формулы

Домен смотрит только на те измерения, которые сам фиксирует.
По каждому такому измерению транзакция должна:

1. иметь значение;
2. попасть в допустимое множество домена.

### Бизнес-смысл

Если домен требует:

- `basis = trx`
- `currency = EUR`

то транзакция:

- `basis = trx, currency = EUR` входит в домен;
- `basis = trx, currency = USD` не входит;
- `basis = rep, currency = EUR` не входит.

Если домен требует:

- `basis = trx`
- `currency = EUR`
- `country in {EE, LV}`

то транзакция с `country = LT` в него уже не входит.

---

### 5.4. Свойства активной системы доменов

#### 5.4.1. Непересекаемость

Для любых двух разных активных доменов $D_1 \ne D_2$:
$$e \in D_1 \Rightarrow e \notin D_2.$$

#### 5.4.2. Полнота

Для любой обслуживаемой транзакции $e$ существует ровно один активный домен $D$, такой что
$$e \in D.$$

### Бизнес-смысл

Активные домены должны образовывать корректное разбиение потока транзакций.

Корректный набор доменов:

- `trx + EUR`
- `trx + USD`
- `rep + EUR`

Некорректный набор доменов:

- `trx + {EUR, USD}`
- `trx + EUR`

потому что любая `trx + EUR` транзакция попадёт сразу в оба домена.

---

## 6. Правило внутри домена

### 6.1. Интуиция

После того как транзакция попала в домен, внутри этого домена применяются правила.

Правило задаёт более узкий бизнес-смысл внутри домена.

---

### 6.2. Scope правила

Правило $R$ задаёт функцию
$$Scope_R(e)\in\{0,1\}.$$

Транзакция участвует в правиле $R$, если
$$e\in D \land Scope_R(e)=1.$$

### Бизнес-смысл

Если домен — это все `trx + EUR`, то rule может выбрать внутри него:

- только `account_type = income`,
- только `account_type = expense`,
- только `country in {EE, LV}`,
- только `vertical = Food`,
- только `legal_entity = Company Ops OÜ`.

Домен задаёт класс суммы, а rule — какой именно balance или агрегат внутри этого класса мы хотим вести.

---

## 7. Materialization

### 7.1. Интуиция

Materialization определяет, по какому ключу balance реально хранится и читается.

Одно правило может иметь одну или несколько materializations.

---

### 7.2. Ключ materialization

Пусть materialization $M$ задана упорядоченным списком измерений
$$GroupBy_M = (g_1,\dots,g_m).$$

Тогда, если все соответствующие значения транзакции определены,
$$Key_M(e) = (e(g_1),\dots,e(g_m)).$$

### Бизнес-смысл

Если `groupBy = (country, vertical)`, то одна и та же rule materialize-ит разные строки для:

- `(EE, Food)`
- `(EE, Grocery)`
- `(LV, Food)`
- `(LV, Grocery)`

Если `groupBy = (country)`, то materialization укрупняется до:

- `EE`
- `LV`
- `LT`

---

### 7.3. Совмещение scope и groupBy

Если измерение используется и в `scope`, и в `groupBy`, то в каноническом пути оно появляется один раз.

При этом:

- `scope` ограничивает допустимые значения;
- `groupBy` делает измерение адресуемой координатой.

### Бизнес-пример

Конфигурация:

```json
{
  "scope": { "country": ["EE", "LV"] },
  "groupBy": ["vertical", "country"]
}
```
означает:
•	в balance входят только EE и LV;
•	внутри balance ключ различается по vertical и country.

То есть country одновременно:
•	ограничивает область действия,
•	и является координатой ключа.означает:
•	в balance входят только EE и LV;
•	внутри balance ключ различается по vertical и country.

То есть country одновременно:
•	ограничивает область действия,
•	и является координатой ключа.

---

## 8. Семейство балансов

### 8.1. Определение

Для фиксированных домена $D$, правила $R$ и materialization $M$ определяется семейство балансов
$$F=(D,R,M).$$

Это множество всех balance rows, которые порождаются одной и той же семантикой:

- одним доменом,
- одним scope,
- одним правилом materialization.

### Бизнес-смысл

Если rule говорит:

- `scope: country in {EE, LV}`
- `groupBy: (vertical, country)`

то family — это не одна строка, а всё множество строк:

- `(Food, EE)`
- `(Food, LV)`
- `(Grocery, EE)`
- `(Grocery, LV)`
- и так далее.

Family — это “тип баланса”, а row — это конкретная ячейка этого типа.

---

### 8.2. Balance row

Balance row — это одна конкретная materialized запись семейства $F$, адресуемая одним полным каноническим путём $p$.

---

## 9. Канонический путь balance row

### 9.1. Интуиция

Чтение должно происходить по одному полному каноническому пути, который включает всю смысловую адресацию строки баланса.

---

### 9.2. Определение

Для семейства $F=(D,R,M)$ и транзакции $e$ канонический путь materialized balance row определяется как
$$BalPath_F(e)=Canon\bigl(DomPath(D), Scope_R, Key_M(e)\bigr),$$
где $Canon$ — операция канонической нормализации пути.

---

### 9.3. Требования к канонизации

Операция $Canon$ обязана:

1. строить путь в глобальном порядке измерений;
2. не дублировать измерения, если они присутствуют в домене, scope и key;
3. сохранять:
    - доменную часть,
    - scope-ограничения,
    - key-значения.

### Бизнес-смысл

Путь чтения должен содержать всю информацию, которая влияет на смысл balance row.

Например, нужно различать:

- `trx / EUR / country=EE / vertical=Food`
- `rep / EUR / country=EE / vertical=Food`

даже если suffix `(country=EE, vertical=Food)` одинаковый.
Различие обеспечивается доменным префиксом.

---

## 10. Баланс семейства

Для семейства $F=(D,R,M)$ баланс по пути $p$ определяется как
$$B_F(p)
=
\sum_{\substack{e\in E\\ e\in D\\ Scope_R(e)=1\\ BalPath_F(e)=p}} Val(e).$$

### Объяснение формулы

В сумму входят все транзакции, которые:

- принадлежат домену,
- удовлетворяют scope правила,
- под materialization порождают один и тот же канонический путь.

### Бизнес-смысл

Если вы ведёте GMV family по `(country, vertical)`, то строка

$$p = (trx,\ EUR,\ country=EE,\ vertical=Food)$$

содержит сумму всех GMV-транзакций EE/Food в transaction-basis EUR.

---

## 11. Current balance

На текущем этапе рассматривается только текущий баланс.

То есть учитывается просто накопленная сумма всех подходящих транзакций. В модель пока не входят:

- обороты по периодам;
- daily, monthly, quarterly buckets;
- snapshots;
- balances as of date.

### Бизнес-смысл

Если сегодня пришли новые транзакции, они просто добавляются в текущую balance row. История по периодам пока отдельно не materialize-ится.

---

## 12. Операция чтения

Чтение определяется по полному каноническому пути:
$$Read(p)=B(p).$$

Если путь $p$ соответствует существующей materialized row, значение возвращается. Иначе — ошибка.

### Требование к сложности

При наличии materialization и индексации чтение должно выполняться за
$$O(1).$$

### Бизнес-смысл

Клиент не должен угадывать конфиг. Он должен знать полный канонический путь.

Примеры путей чтения:

- `trx / EUR / expense / ops_cost / EE / Company Ops OÜ`
- `rep / EUR / GMV / country=LV / vertical=Food`

Если такой path поддерживается системой, balance читается напрямую.

---

## 13. Уникальность путей внутри домена

### 13.1. Интуиция

Чтобы чтение было однозначным, разные активные families внутри одного домена не должны порождать пересекающиеся пространства путей.

---

### 13.2. Path-space-disjointness

Для двух разных семейств $F_1=(D,R_1,M_1)$ и $F_2=(D,R_2,M_2)$ внутри одного домена $D$ должно выполняться:
$$PathSpace(F_1)\cap PathSpace(F_2)=\varnothing.$$

### Бизнес-смысл

Внутри одного домена нельзя иметь два активных rules/materializations, которые могут породить один и тот же balance path.

Недопустимый пример:

Rule 1:
```json
{
  "scope": { "country": ["EE", "LV"] },
  "groupBy": ["vertical", "country"]
}
```

Rule 2:
```json
{
  "scope": { "country": ["EE", "LT"] },
  "groupBy": ["vertical", "country"]
}
```
Оба rules могут породить путь (country=EE, vertical=Food), значит одновременно активными быть не должны.

---

## 14. Аддитивность

Аддитивность гарантируется в рамках одного фиксированного семейства $F$.

Если $E_1$ и $E_2$ — два множества транзакций, то
$$B_F^{E_1\cup E_2}(p)=B_F^{E_1}(p)+B_F^{E_2}(p).$$

### Бизнес-смысл

Если у вас есть family “GMV by country and vertical”, то оно аддитивно по транзакциям: можно считать часть данных отдельно, а потом сложить результаты.

Это свойство относится именно к одному family, а не ко всему домену автоматически.

---

## 15. Структурные свойства доменов

Над доменами корректно определять структурные свойства, а не вычислительные операции.

### 15.1. Равенство доменов

Два домена равны, если совпадают их доменные шаблоны.

### 15.2. Непересекаемость доменов

Домены не пересекаются, если не существует транзакции, принадлежащей обоим.

### 15.3. Полнота системы доменов

Система доменов полна относительно входного потока, если любая обслуживаемая транзакция попадает ровно в один домен.

### Бизнес-примеры

Корректная система:

- `trx + EUR`
- `trx + USD`
- `rep + EUR`

Некорректная система:

- `trx + {EUR,USD}`
- `trx + EUR`

потому что `trx + EUR` транзакции попадают в оба домена.

---

## 16. Операции над семействами

Вычислительные операции определяются не над доменами, а над families.

### 16.1. Conversion

Conversion — унарная операция, переводящая одно family в другое.

Пусть есть families
$$F_A,\quad F_B.$$

Conversion задаётся:

- отображением путей
  $$\Psi : PathSpace(F_A)\to PathSpace(F_B),$$
- правилом численного преобразования
  $$\Gamma : \mathbb R \to \mathbb R$$
  или более общо $\Gamma_{p_A\to p_B}$.

Тогда
$$Conv_{F_A\to F_B}(p_B)
=
\sum_{\substack{p_A\in PathSpace(F_A)\\ \Psi(p_A)=p_B}}
\Gamma_{p_A\to p_B}\bigl(B_{F_A}(p_A)\bigr).$$

### Бизнес-смысл

Это общий механизм перевода balances.

Примеры:

- из детального balance в более грубый;
- из `trx`-семейства в `rep`-семейство, если есть корректное правило преобразования;
- из одной схемы ключа в другую.

---

### 16.2. Roll-up как частный случай conversion

Roll-up — это conversion, в котором:

- домен не меняется;
- числовое преобразование тождественно:
  $$\Gamma(x)=x;$$
- path mapping many-to-one схлопывает более детальный путь в более грубый.

### Бизнес-пример

Если есть family:

- `GMV by (country, vertical)`

и family:

- `GMV by (country)`

то
$$GMV(country)=\sum_{vertical} GMV(country, vertical)$$

это roll-up, и он же частный случай conversion.

---

### 16.3. Derived composition

Derived composition — это операция над несколькими families:
$$Der_\Phi(F_1,\dots,F_n)=F_*.$$

На уровне значений:
$$B_{F_*}(p)
=
\Phi\bigl(B_{F_1}(p),\dots,B_{F_n}(p)\bigr).$$

### Бизнес-примеры

1. Прибыль:
   $$Profit(country)=Income(country)-Expense(country)$$

2. Исправленный balance:
   $$Corrected(p)=Base(p)+Adjustment(p)$$

3. Суммарный GMV:
   $$TotalGMV = GMV_{Food} + GMV_{Grocery}$$

### Почему это не conversion

Если
$$F_1 + F_2 = F_3$$
то $F_3$ зависит от двух источников, значит это composition, а не унарное преобразование.

---

## 17. Что не входит в формальное ядро

Следующие вещи не являются базовыми формальными операциями текущей теории.

### 17.1. Replacement

Это не математическая операция, а policy/versioning: какой объект считать активным источником истины.

### 17.2. Correction

Это либо:

- обычное добавление новых транзакций,
- либо частный случай derived composition.

Поэтому `replacement` и `correction` не входят в формальное ядро.

## 18. Бизнес-примеры

### 18.1. Бухгалтерский balance по счёту

Пусть домен:

- `basis = trx`
- `currency = EUR`

Rule:

- `account_type = expense`
- `account_subtype = ops_cost`

Materialization:

- `groupBy = (country, legal_entity)`

Тогда family ведёт строки вида:

- `trx / EUR / expense / ops_cost / EE / Company Ops OÜ`
- `trx / EUR / expense / ops_cost / LV / Company Technology OU`

Это бухгалтерский balance по счёту, уточнённый аналитикой.

---

### 18.2. GMV по странам и вертикалям

Пусть домен:

- `basis = trx`
- `currency = EUR`

Rule:

- completed GMV events

Materialization:

- `groupBy = (country, vertical)`

Тогда family ведёт строки:

- `trx / EUR / country=EE / vertical=Food`
- `trx / EUR / country=LV / vertical=Grocery`

Это GMV-balance family.

---

### 18.3. Ограниченный scope по Балтике

Rule:

```json
{
  "scope": { "country": ["EE", "LV"] },
  "groupBy": ["vertical", "country"]
}
```

Это означает:

- учитываются только EE и LV;
- внутри family строки различаются по `vertical` и `country`.

Строк для `LT` не будет.

---

### 18.4. Недопустимое пересечение rules

Rule 1:

```json
{
  "scope": { "country": ["EE", "LV"] },
  "groupBy": ["vertical", "country"]
}
```

Rule 2:

```json
{
  "scope": { "country": ["EE", "LT"] },
  "groupBy": ["vertical", "country"]
}
```

Они оба могут породить:

- `(Food, EE)`
- `(Grocery, EE)`

Значит в одном домене такие rules одновременно активными быть не должны.

---

### 18.5. Conversion `trx -> rep`

Пусть есть два families с одинаковым suffix key `(country, vertical)`, но в разных доменах:

- `trx / EUR / country / vertical`
- `rep / EUR / country / vertical`

Если есть корректное правило преобразования значений и path mapping между ними, conversion допустима.

Если же для перехода нужны данные, потерянные при агрегации, conversion недопустима.

---

### 18.6. Derived profit

Есть families:

- `income_by_country`
- `expense_by_country`

Тогда derived family:

$$profit_by_country = income_by_country - expense_by_country$$

Это не conversion, а composition двух balances.

## 19. Что изменится, если отказаться от непересекаемости доменов

Теория выше предполагает непересекаемость доменов. Это даёт очень сильный инвариант:

> каждая транзакция принадлежит ровно одному классу аддитивности.

Если от этого предположения отказаться, система станет гибче, но теория существенно усложнится.

### 19.1. Что появляется

Если домены разрешено пересекать, одна транзакция может принадлежать нескольким доменам одновременно.

Тогда домены перестают быть разбиением и становятся покрытием.

### Бизнес-смысл

Это может быть полезно, если одна и та же транзакция должна одновременно участвовать в нескольких независимых представлениях:

- бухгалтерское представление,
- reporting-представление,
- managerial slice,
- GMV-представление.

### 19.2. Что теряется

Если непересекаемость убрать, теряются следующие сильные свойства.

#### 19.2.1. Однозначная доменная классификация

Вместо
$$e \mapsto D$$
получается
$$e \mapsto \{D_1,\dots,D_k\}.$$

#### 19.2.2. Уникальный класс аддитивности

Транзакция больше не находится в одном уникальном классе суммы. Она принадлежит сразу нескольким семантическим представлениям.

#### 19.2.3. Безопасность композиции по умолчанию

Если одна и та же транзакция попала:

- в family $F_1$ домена $D_1$,
- и в family $F_2$ домена $D_2$,

то операции вида:

- сумма,
- derived composition,
- conversion,

могут посчитать одну и ту же экономическую массу повторно.

То есть возникает риск двойного учёта.

#### 19.2.4. Неоднозначность conversion

Если транзакция уже имеет native-представление в двух доменах, то conversion из одного в другой уже не обязательно означает настоящий перевод. Это может быть просто второе представление уже существующей массы.

### 19.3. Что остаётся корректным

Даже без непересекаемости:

- чтение по полному каноническому пути может оставаться технически однозначным;
- аддитивность внутри одного фиксированного family сохраняется.

То есть локальный закон
$$B_F^{E_1\cup E_2}(p)=B_F^{E_1}(p)+B_F^{E_2}(p)$$
остаётся верным для одного family.

### 19.4. Какие новые требования тогда нужны

Если домены пересекаются, корректность уже нельзя обеспечивать только через устройство доменов. Её нужно переносить на уровень операций над families.

Минимально потребуется учитывать:

- какие исходные транзакции contributed to each family;
- пересекаются ли source sets двух families;
- допустима ли композиция при таком пересечении.

В частности:

#### Для derived sum

Если определено
$$F_1 + F_2,$$
то должно выполняться либо
$$Src(F_1)\cap Src(F_2)=\varnothing,$$
либо должно быть явно задано правило коррекции пересечений.

#### Для conversion

Нужно различать:

- native target representation,
- converted target representation.

Иначе одна и та же масса может появиться дважды.

#### Для roll-up

Roll-up безопасен только внутри одной lineage-связки, то есть когда грубый family получается из более детального family, представляющего ту же source mass.

### 19.5. Концептуальное изменение

При непересекаемости домены являются:

- классами аддитивности,
- и разбиением транзакций.

Без непересекаемости домены превращаются скорее в:

- семантические линзы,
- или представления,

а не в уникальные классы владения суммой.

Тогда бремя корректности переносится с доменов на операции над families.

### 19.6. Практический вывод

Система с непересекающимися доменами:

- проще для reasoning;
- даёт меньше риска двойного учёта;
- упрощает conversion и derived balances.

Система с пересекающимися доменами:

- гибче;
- но требует lineage tracking;
- overlap checks;
- условий допустимости для composition;
- явного контроля семантического дублирования.

### 19.7. Итог

Для базового ядра теории предпочтительно сохранять непересекаемость доменов.

Это даёт сильный инвариант, который упрощает:

- membership in domain,
- additivity,
- family construction,
- conversion,
- derived composition,
- semantic reasoning.

Теория пересекающихся доменов возможна, но должна рассматриваться как отдельное последующее расширение, а не как базовое ядро.
